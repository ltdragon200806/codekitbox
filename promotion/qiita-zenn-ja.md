# Qiita / Zenn 投稿用記事

## おすすめタイトル

JWT の中身を安全に確認するためのブラウザツールを作った

## 本文

API の開発やデバッグでは、JWT の `header` や `payload` をすぐに確認したい場面が
よくあります。

JWT は Base64URL 形式でエンコードされているため、内容を確認するだけなら難しい処理
は必要ありません。しかし、本番環境のトークンをよく知らないオンラインサービスへ
送信することには不安があります。

そこで、一般的な入力を可能な限りブラウザ内で処理する無料ツール集
**CodeKitBox** を作りました。

https://codekitbox.com/ja/tools/jwt-decoder/

登録は不要で、JWT デコーダー以外にも JSON フォーマッター、Unix タイムスタンプ
変換、Base64、QR コード生成などを利用できます。

## JWT の基本構造

JWT は通常、次の3部分で構成されています。

```text
header.payload.signature
```

- `header`: 署名アルゴリズムやトークンの種類
- `payload`: ユーザーや有効期限などのクレーム
- `signature`: 内容が変更されていないことを検証するための署名

デコードツールでは、主に `header` と `payload` を読みやすい JSON として表示します。

## 確認することが多いクレーム

JWT のデバッグでは、次の値を確認することが多いです。

- `iss`: トークンの発行者
- `aud`: トークンの対象
- `sub`: ユーザーなどの主体
- `iat`: 発行時刻
- `nbf`: 利用開始時刻
- `exp`: 有効期限

`iat`、`nbf`、`exp` は通常 Unix タイムスタンプの秒単位です。ミリ秒として処理すると、
大きく異なる日時になるため注意が必要です。

## デコードと検証は別の処理

最も重要な点は、JWT をデコードできても、そのトークンが正しいとは限らないことです。

`header` と `payload` は秘密情報ではなく、署名鍵を持っていない人でも読むことが
できます。そのため、アプリケーションでは必ず次の処理が必要です。

1. 信頼できる鍵で署名を検証する
2. 期待する署名アルゴリズムか確認する
3. `iss` と `aud` が正しいか確認する
4. `exp` と `nbf` を確認する
5. アプリケーション独自の権限ルールを確認する

CodeKitBox の JWT デコーダーは内容確認用です。署名検証や認証判断を行うものでは
ありません。

## なぜブラウザ内で処理するのか

オンラインツールは便利ですが、API レスポンス、ログ、アクセストークンなどを
貼り付ける場合があります。

CodeKitBox は HTML、CSS、JavaScript で構成された静的サイトです。JWT、JSON、
Base64、通常のテキストなど、一般的な入力は可能な限りブラウザ内で処理します。

ただし、ブラウザ内で処理される場合でも、本番環境の重要なトークンを必要なく共有
することは避けるべきです。安全なテスト用トークンを使用することをおすすめします。

- Unix タイムスタンプ変換
- UUID、SHA ハッシュ、QR コード、パスワード生成
- 正規表現テスト、テキスト比較、文字数カウント
